gRPC QA Test Lab

Minimal gRPC service for QA testing, contract validation, and learning microservice communication.

📦 Services

UserService BlacklistService

This system contains two microservices communicating via gRPC.

🌐 Endpoints

UserService → localhost:50051 BlacklistService → localhost:50052

No HTTP endpoints. Communication is protocol-buffer based (gRPC).

📡 Methods (RPC Calls)

UserService: GetUser(UserRequest) returns (UserResponse) BlacklistService: CheckBlacklist(BlacklistRequest) returns (BlacklistResponse)

📨 Messages (Schema)

UserRequest: email (string) UserResponse: email (string) status (string) BlacklistRequest: email (string) BlacklistResponse: blocked (bool)

🧪 How to Test

Step 1: Import testlab.proto into Postman (gRPC) Step 2: Select method: UserService / GetUser Step 3: Request: { "email": "bad@email.com" } Step 4: Expected response: { "email": "bad@email.com", "status": "BLOCKED" }

⚙️ Service Behavior

If email is in blacklist: → BLOCKED If email is NOT in blacklist: → ACTIVE Validation is handled by BlacklistService via internal gRPC call.

⚠️ Negative Scenarios

Empty email: → validation error Service unavailable: → UNAVAILABLE error Invalid request structure: → proto mismatch error

📚 QA Focus

- Contract-based testing (.proto) - Microservice dependency validation - gRPC communication flow - Data integrity across services
